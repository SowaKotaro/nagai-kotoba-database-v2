# 単語の補完・補足（候補の一括収集・オフライン）

nagai-kotoba-database-v2 に**まだ登録されていない単語の候補を増やす**ための、アプリから切り離した調査タスク。
既に登録済みの語（種語）を1つ受け取り、その**上位概念**を突き止め、**同じ系統の語をまとめて刈り取る**。
`research/` 配下の入力ファイルを読み、調査結果を出力ファイルに書き出す（**ファイル入力 → 調査 → ファイル出力**）。

これはパイプラインの**0段目**（候補収集）。出力の語リストはそのまま `/notation`（表記）→ `/reading`（読み）
→ 登録 → `/annotation`（注釈）へ流す。

## フロー
1. **入力**: `research/inputs/expansion.txt` を読む。1行＝1語（種語）。
   行頭の番号/記号（`1.` `-` `・`）は無視。空行は無視。表層形はスペースを含みうる（例「Dead by Daylight」）＝**行全体を1語**として扱う。
2. **調査**: 種語ごとに 上位概念 → 収集軸の選定 → 同系統の語 の順に WebSearch で調べる（下記）。
3. **出力**: `research/outputs/expansion.txt` に書き出す（**上書き**）。

## 調べること / 調べないこと
- **調べる**: 種語と**同じ系統の未登録語（候補）を列挙すること**だけ。
- **調べない**: 表記の正規化（→ `word-notation-research`）、読み（→ `word-reading-research`）、
  意味・ジャンル・品詞・語種（→ `word-annotation-research`）。**候補の表層形を挙げるところまでが仕事**。

## 大原則
- **判断基準は [`docs/annotation-guidelines.md`](../../../docs/annotation-guidelines.md) が正。**
  収録の4原則（§1: 公然性・妥当性・慣用性・同定可能性）を満たさない語は**候補に入れない**。
  知られているかどうかの判定は §5（知名度・一般性の判定目安）に従う。
- **でっち上げない（最重要）。** このスキルは「知らない語を大量に出す」ため、**幻覚が最も出やすい**。
  公式サイト・Wikipedia の一覧記事など**実在する一覧を情報源として確認**し、
  そこに載っている語だけを挙げる。一覧が見つからない／断片的にしか確認できない場合は、
  **数を埋めようとせず、確認できた分だけを出して `notes` に「網羅できていない」と書く**。
  「たぶんこういう名前の技もあったはず」で書かない。
- **記憶で書かず、必ず WebSearch する。** 種語がどれだけ有名でも、収集の根拠は毎回引き直す。

## 手順

### 1. 上位概念を特定し、収集軸を選定する
種語を WebSearch し、まず**上位概念**を確定する。

- **上位概念**: その語が属する作品・アーティスト・シリーズ・組織・ブランド・体系など。
  （例: `泥門デビルバッツ` → 漫画『アイシールド21』）

次に、その上位概念から取れる**収集軸**（エンティティの軸）を洗い出し、
**最も有望な軸を1つ選ぶ**。（例: アイシールド21 なら チーム名／キャラクター名／必殺技名）
種語自身の軸（泥門デビルバッツならチーム名）は有力候補だが、自動的には選ばない。
軸の有望さは次で判断する。

- **長い語が見込めるか**: 読み10文字以上になりそうな語が多い軸を優先する。
  短い語ばかりの軸は、一覧がどれだけ充実していても選ばない。
- **実在する一覧で裏が取れるか**: 公式サイト・Wikipedia の一覧記事など、
  出典にできる一覧が実在する軸を優先する。
- **収録4原則（guidelines §1）を満たす語が見込めるか**。

**1つの種語につき選ぶ軸は1つ**。軸を混ぜると出力のジャンル・エンティティがバラバラになり、
後工程が破綻する。選ばなかった有望軸は収集せず、`notes` に「この軸も取れる」と提案するに留める
（オーナーが次回 `/expand` の引数で指定できる）。

**コマンド引数で軸が指定された場合は、選定を行わずその軸で収集する**（例: `/expand キャラクター名` なら、
有望さの判断を待たずキャラクター名を集める）。

**有望な軸が無い種語は、無理に拡張しない。** どの軸も短い語ばかり／一覧の裏が取れない／
4原則を満たす語が見込めない、という種語は**収集0件でよい**。
理由を `--- 注記 ---` に書いて次の種語へ進む。数を出すことより、出した語が全部使えることのほうが大事。

### 2. 同系統の語を列挙する
上位概念 × 収集軸 の一覧を情報源から取る。公式サイトの登場人物/ディスコグラフィ一覧、
Wikipedia の作品記事・一覧記事などを当たり、**出典URLを控える**（出力の `--- 系統 ---` に書く）。

- **上限は 1種語 × 1軸あたり 30語**。超える場合は**より一般に知られているもの**から30語を選び、
  `notes` に「他に○件あり」と件数を明記する（オーナーが必要なら次回続きを取る）。
- **種語自身は出力に含めない**（既に登録済みのはず）。
- **既に DB に登録済みかどうかは調べない**（このスキルは DB を見ない）。重複は登録 step3 の
  重複チェックが弾くので、気にせず候補として出してよい。

### 3. 候補をふるいにかける
次のいずれかに当たる語は、上部の候補リストに入れず `--- 除外した語 ---` に理由付きで落とす。
**捨てずに残す**のは、オーナーが判断を覆して復活させられるようにするため。

- **明らかに短い語**。収録のボーダーは**読み10文字以上**だが、この段階では読みが未確定なので
  **厳密には見ない**。カタカナに直しておおよそ10モーラに**遠く届かないもの**（例: `セナ`、`ヒル魔`）だけを弾き、
  **際どいもの（8〜12モーラ程度）は残して**判定を後工程の `/reading` に委ねる。**迷ったら残す。**
- **収録4原則（guidelines §1）を欠く語**。作中に1度しか出てこない語、公式に名前が確認できない通称、
  独立した情報源で裏が取れない語など。

## 出力フォーマット（`research/outputs/expansion.txt`）

先頭に、収集した候補語を**通し番号付きで1行1語**並べる。
**この上部リストはそのままコピペして `/notation` の入力（`research/inputs/notation.txt`）に使うので、
表層形そのものだけを書く**（`（アイシールド21）` などの注記や補足を行末に付けない）。
表記の正規化は次工程の仕事なので、ここでは**情報源に載っている表記のまま**でよい。

続けて次の3節を置く。

- `--- 系統 ---`: 種語ごとに 上位概念 / 収集軸（**選定理由も一言**）/ 出典URL / 収集件数。
- `--- 除外した語 ---`: `<語>（除外理由）`。無ければこの節は省く。
- `--- 注記 ---`: 種語ごとに confidence（high / medium / low）、網羅できたかどうか、
  上限で切った件数、**他に取れる軸の提案**。**拡張を見送った種語**もここに理由を書く。

例:

```
# 単語の補完・補足（research/inputs/expansion.txt の調査結果）
1. 神龍寺ナーガ
2. 白秋ダイナソアーズ
3. 帝黒アレキサンダーズ
4. 盤戸スパイダーズ
5. 巨深ポセイドン

--- 系統 ---
泥門デビルバッツ
  上位概念: 漫画『アイシールド21』（稲垣理一郎・村田雄介）
  収集軸: チーム名（「地名＋ニックネーム」構成で長い語が多く、Wikipedia に一覧があるため選定）
  出典: https://ja.wikipedia.org/wiki/アイシールド21
  収集: 5件（除外2件）

--- 除外した語 ---
王城ホワイトナイツ（読みが確定すれば10文字を超える見込みだが、公式一覧で表記が割れるため要確認）
NASA エイリアンズ（欧文表記が混在。表記確定が必要）

--- 注記 ---
泥門デビルバッツ: confidence: high。Wikipedia の登場チーム一覧から取得。主要チームは網羅できている。
  上限（30語）には達していない。
  他に取れる軸: キャラクター名（例: 小早川瀬那ら）、必殺技名（例: デビルバットゴースト）。
  必要なら `/expand キャラクター名` で再実行できる。
初恋クレイジー: 収集0件（拡張見送り）。上位概念はスピッツのアルバム『インディゴ地平線』。
  収録曲名の軸は短い曲名が大半で、読み10文字以上が見込める語がほぼ無い。
  アルバム名の軸も同様に短い。有望な軸が無いため無理に拡張しない。
```

## この後の流れ
候補リスト（`research/outputs/expansion.txt` 上部）は、そのまま `word-notation-research` の入力
`research/inputs/notation.txt` に渡す（**補完 → 表記 → 読み → 登録 → 注釈**）。
表記の正規化・読みの確定・重複チェックはすべて後工程が担う。
