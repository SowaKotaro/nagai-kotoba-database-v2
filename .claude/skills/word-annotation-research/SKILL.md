# 単語のアノテーション調査（オフライン）

nagai-kotoba-database-v2 のアノテーション作業を助けるための、**アプリから切り離した**調査タスク。
管理画面の「調査用データの書き出し」が出力する JSON（対象語 + マスタ一覧）を受け取り、各語について
下記スキーマの **提案 JSON だけ** を出力する。この JSON を管理画面の「提案 JSON の取り込み」に貼ると
下書き（annotation_proposals）として保存され、オーナーがコンソールで1語ずつ確認・修正して承認する。

## 大原則
- **判断基準は [`docs/annotation-guidelines.md`](../../../docs/annotation-guidelines.md) が正。**
  収録の4原則（公然性・妥当性・慣用性・同定可能性）・立項スコアの定義・表記/読み/知名度の基準は
  必ずそこに従う。オーナー個人の感覚ではなく、この基準で判定することが DB のバイアス防止になる。
- **提案はあくまで下書き。** 公開データには直接入らないので、書けることは全部書く。ただし
  **不確実な内容をでっち上げない**。迷ったら `confidence: "low"` にして `notes` に迷いの内容を書く。
- **マスタは渡された一覧から選ぶ。** `masters` にある名前と**一字一句同じ表記**で書く（取り込み側は
  名前でマスタを解決する）。適切な選択肢が無いときだけ新しい名前を書き、ジャンルなら `genre_new: true`
  を立てる（新設候補としてオーナーに提示される）。
- **固有名詞・作品名・新語は WebSearch で裏取りする。** 意味の記述・ジャンル判定・エンティティ判定の
  根拠を最低1つの信頼できる情報源で確認する。裏取りできなければ confidence を下げる。

## 入力（ファイル）
リポジトリ直下 `inputs_annotation_tmp.json` を読む。中身は管理画面の「調査用データの書き出し」が出力する
JSON（対象語 + マスタ一覧）を保存したもの:
- `words`: 調査対象。`word_id`（取り込みのキー。**そのまま返す**）・`surface`（表層形）・`reading`（読み）
- `masters`: 選択肢。`genres`（大→中→小のパスの配列）・`entity_types`・`parts_of_speech`・
  `word_origins`・`linguistic_features`（いずれも名前の配列）

## 各語は「2周」で調べる
精度を上げるため、**1語につき調査を2周**行う。
1. **1周目（収集）**: WebSearch で意味・ジャンル・エンティティ・品詞・語種・別表記・立項根拠を集める。
   この段階では広く集めてよい。
2. **2周目（ファクトチェック）**: 1周目で書いた各項目を**別の検索・別の情報源で裏取りし直す**。
   - 意味の記述に事実誤認が無いか（人物・年号・作者・所属など）を、独立した情報源で確認する。
   - 表記（最も一般的な表記か）・読み・ジャンルの当てはめ・エンティティ種別を再確認する。
   - 1周目と食い違ったら、より信頼できる情報源に合わせて直し、迷いが残れば `confidence` を下げて
     `notes` に「何を確認し直したか・何が確定できなかったか」を書く。
   - 2周して裏が取れた内容だけを `high` にする。1周のみ・伝聞のみは `medium` 以下にする。

## 調査手順（1語ずつ）
1. 表層形と読みから語を特定する。曖昧な語（同名の作品・人物が複数あるなど）は WebSearch で確定し、
   どれを採ったか `notes` に書く。
   - **同じ表層形で複数の異なる語義（同名の作品・人物・団体等）が無いか毎語 WebSearch で確認する**
     （例:「ピーターパンシンドローム」は通俗心理学の用語と、同名の男性アイドルグループの両方が存在する）。
   - **複数の語義が見つかったら `senses` 配列で複数の語義を返す**（同音異義語対応・Issue 41）。
     各要素が `meaning`/`genre_path`/`entity_type`/`part_of_speech`/`word_origins`/`variants` を持つ。
     取り込み側は語義ごとに `word_sense` を組み立てる。語の読みが語義で共通なら各 `sense` の `reading`
     は省略してよい（語義で読みが変わるときだけ入れる）。慣用度の高い語義から順に並べる。
   - 単一語義なら従来どおり `meaning` などをトップレベルに置いてよい（`senses` を使うなら重複させない）。
2. **立項判定（`entry_score`）**: guidelines の収録4原則（公然性・妥当性・慣用性・同定可能性）に
   照らして 1〜5 のスコアを**全語に**付ける。
   - 5=4原則すべて明確に満たす（辞書・百科事典・公式媒体で即確認）／ 4=満たすが裏取りが薄い ／
     3=1原則にグレー（オーナー判断）／ 2=1原則を明確に欠く ／ 1=複数原則を欠く（内輪語・即興造語・検証不能）
   - **3以下には `entry_notes` を必ず書く**（どの原則を・なぜ欠くか）。登録済みの語への低スコアは
     削除候補の指摘になる（削除の最終判断はオーナー）。
   - スコアは「有名さ」ではなく基準への適合度。マイナーでも4原則を満たせば5になり得る。
3. **意味（`meaning`）**: 公開ページのリード文になり得るトーンで、1〜2文・自己完結で書く。
   文体は「〜。」の常体（例:「漫画『ONE PIECE』の登場人物。麦わらの一味の船長。」）。
   表記に疑義がある場合（冠称付き・説明句・異体字など guidelines §3 に反する表層形）は
   `notes` で指摘する。
4. **ジャンル（`genre_path`）**: `masters.genres` のパスから最も適切な**末端（小分類）まで**を選ぶ。
   適切な小分類が無ければ、既存の大・中分類の下に新しい小分類名を置き `genre_new: true` を立てる。
5. **エンティティ（`entity_type`）**: 固有名詞なら該当する種類を `masters.entity_types` から。
   一般語なら省略する。
6. **品詞（`part_of_speech`）**: `masters.parts_of_speech` から。ほとんどの収録語は名詞・固有名詞系。
7. **語種（`word_origins`）**: 和語・漢語・外来語・混種語などを `masters.word_origins` から（複数可）。
   読みと表記から機械的に判定できることが多い（カタカナ外来語、漢字熟語=漢語 など）。
8. **別表記（`variants`）と最も一般的な表記**: まず**その語の最も一般的な表記を WebSearch で確定する**
   （辞書見出し・公式表記・用例数などで判断）。その上で、一般に流通する**それ以外の表記**を毎語
   確認して `variants` に記録する（正式名称と通称、カタカナ表記とその和訳、辞書見出しの表記ゆれ、
   送り仮名・漢字/かなの揺れなど）。各 `variant` は `surface`（+読みが変わるなら `reading`）で書く。
   - 取り込み対象の表層形（`word_id` に紐づく登録済み `surface`）が**最も一般的な表記と違う**場合は、
     `surface` を変えることはできない（登録時に確定済み）ので、`notes` に「最も一般的な表記は○○」と
     明記し、一般的な表記を `variants` にも入れてオーナーの判断（表層形の訂正）を促す。
   - 別表記が無ければ省略してよい。
9. `confidence` を付ける: `high`=裏取り済みか定番 / `medium`=妥当だが未検証 / `low`=不確実。
   `low` のときは必ず `notes` に「何に迷ったか・何を確認してほしいか」を書く。

## 出力（ファイル）
- リポジトリ直下 `outputs_annotation_tmp.json` に、**下記スキーマの JSON だけ**を書き出す（**上書き**。説明文は入れない）。
  このファイルの中身をそのまま管理画面の「提案 JSON の取り込み」に貼れる。
- `word_id` は入力の値をそのまま返す（取り込みの突き合わせキー）。
- 値の無いキーは省略してよい。スキーマの定義と例は同ディレクトリの
  [`schema.json`](schema.json) / [`example.json`](example.json) を参照。
