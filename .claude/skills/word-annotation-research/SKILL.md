# 単語のアノテーション調査（オフライン）

nagai-kotoba-database-v2 のアノテーション作業を助けるための、**アプリから切り離した**調査タスク。
管理画面の「調査用データの書き出し」が出力する JSON（対象語 + マスタ一覧）を受け取り、各語について
下記スキーマの **提案 JSON だけ** を出力する。この JSON を管理画面の「提案 JSON の取り込み」に貼ると
下書き（annotation_proposals）として保存され、オーナーがコンソールで1語ずつ確認・修正して承認する。

## 大原則
- **判断基準は [`docs/annotation-guidelines.md`](../../../docs/annotation-guidelines.md) が正。**
  収録の4原則（公然性・妥当性・慣用性・同定可能性）・立項スコアの定義・表記/読み/知名度の基準は
  必ずそこに従う。オーナー個人の感覚ではなく、この基準で判定することが DB のバイアス防止になる。
- **提案はあくまで下書き。** 公開データには直接入らないので、書けることは全部書く。ただし
  **不確実な内容をでっち上げない**。迷ったら `confidence: "low"` にして `notes` に迷いの内容を書く。
- **マスタは渡された一覧から選ぶ。** `masters` にある名前と**一字一句同じ表記**で書く（取り込み側は
  名前でマスタを解決する）。適切な選択肢が無いときだけ新しい名前を書き、ジャンルなら `genre_new: true`
  を立てる（新設候補としてオーナーに提示される）。
  **このファイルの例に出てくるマスタ名は説明用**であり、実データとは限らない。**必ず `masters` を正とする**
  （例: 品詞は「名詞」ではなく `普通名詞`/`固有名詞`、語種は「外来語」ではなく `英語` のように言語名で持つ）。
- **固有名詞・作品名・新語は WebSearch で裏取りする。** 意味の記述・ジャンル判定・エンティティ判定の
  根拠を最低1つの信頼できる情報源で確認する。裏取りできなければ confidence を下げる。
- **書きぶり・粒度に迷ったら、承認済みの公開データに倣う。** 公開 API
  （`https://www.nagai-kotoba-database.jp/words.json` → `/words/<id>.json`）でオーナー承認済みの
  実例を数件参照し、意味の文体・ジャンルの深さ・エンティティの名付けを揃えてよい（任意）。

## ジャンルとエンティティを混同しない（最重要）

この2つは軸が違う。取り違えると分類木が壊れるので、毎語かならず切り分ける。

- **ジャンル（`genre_path`）= 所属**。その語が**どの分野に属するか**。木を何段降りても最後まで「所属」であって、
  語の正体を言い当てるものではない。
- **エンティティ（`entity_type`）= 正体**。その語が**一体何であるか**。人名・作品名・出来事・グループ名 など。

判定は、次の2文が**両方とも自然に読めるか**で確かめる。

> 「ボストン茶会事件」は **出来事** である。（エンティティ）
> 「ボストン茶会事件」は **歴史・地理 > 北アメリカ史・中央アメリカ史 > アメリカ独立革命** に属する。（ジャンル）

よくある取り違え:

| アンチパターン | なぜ駄目か | 正しくは |
|---|---|---|
| `genre_path` の末端に「出来事」「作品」「用語」 | それは正体であって所属ではない | `entity_type` へ |
| `entity_type` に「歴史」「心理学」 | それは所属であって正体ではない | `genre_path` へ |
| `genre_path` の末端に作品名そのもの（例 `ONE PIECE`） | 個体名は分類の階層ではない | 所属は「漫画・コミック」等。作品名は `entity_type`/語自体で表す |

`genre_path` と `entity_type` は、**品詞に関わらず全語に付ける**。所属も正体も必ず決まるからで、
普通名詞だからといって省略しない。承認済みの実例: `エルニーニョ現象`（普通名詞）の正体は `現象名`、
`パラジクロロベンゼン` は `物質名`、`ピーターパンシンドローム` は `俗称症候群`。

**ジャンルの末端とエンティティが同じ名前になったら、たいていどちらかが間違っている。** ジャンル末端には
**分野名**が来るはずで、正体（種類）が紛れ込んでいることが多い。例えば `有機化合物`（正体）をジャンル末端に
置きたくなったら、そこは分野名の `有機化学` が正しく、語の正体は `物質名` としてエンティティへ回す
（承認済み実例: パラジクロロベンゼン = `自然科学 > 化学 > 有機化学` × `物質名`）。

ただし `ことわざ` のように、**分野の区分そのものが語の正体と一致する**語類もある（`成句論` の下位区分は
諺・慣用句・故事成語という「種別」なので、所属の末端も正体も `ことわざ` になる）。この場合は同名でよいが、
`notes` に一言断る。

## 入力（ファイル）
`research/inputs/annotation.json` を読む。中身は管理画面の「調査用データの書き出し」が出力する
JSON（対象語 + マスタ一覧）を保存したもの:
- `words`: 調査対象。`word_id`（取り込みのキー。**そのまま返す**）・`surface`（表層形）・`reading`（読み）
- `masters`: 選択肢。`entity_types`・`parts_of_speech`・`word_origins`・`linguistic_features` は名前の配列。
  `genres` だけは3階層の入れ子オブジェクト **`{大分類: {中分類: [小分類, ...]}}`**:
  ```json
  { "歴史・地理": { "アジア史・東洋史": ["古代文明"], "北アメリカ史・中央アメリカ史": [] } }
  ```
  - 小分類（配列の要素）はそのまま選べる末端。`genre_path` は木を大→中→小と辿った名前3つで組み立てる
    （例 `["歴史・地理", "アジア史・東洋史", "古代文明"]`）。
  - 中分類（内側のキー）は小分類を新設するときの**寄せ先**。配列が空 `[]` でも寄せ先として使える。
  - **`masters.genres` を最初に通読し、既存の中分類の切り口（分割軸）を把握してから**各語を判定する。

## 各語は「2周」で調べる
精度を上げるため、**1語につき調査を2周**行う。
1. **1周目（収集）**: WebSearch で意味・ジャンル・エンティティ・品詞・語種・言語的特徴・別表記・
   立項根拠を集める。この段階では広く集めてよい。
2. **2周目（ファクトチェック）**: 1周目で書いた各項目を**別の検索・別の情報源で裏取りし直す**。
   - 意味の記述に事実誤認が無いか（人物・年号・作者・所属など）を、独立した情報源で確認する。
   - 表記（最も一般的な表記か）・読み・ジャンルの当てはめ・エンティティ種別を再確認する。
   - 1周目と食い違ったら、より信頼できる情報源に合わせて直し、迷いが残れば `confidence` を下げて
     `notes` に「何を確認し直したか・何が確定できなかったか」を書く。
   - 2周して裏が取れた内容だけを `high` にする。1周のみ・伝聞のみは `medium` 以下にする。

## 出力の形式: 常に `senses` 配列で返す

語義の中身（意味・ジャンル・エンティティ・品詞・語種・言語的特徴・別表記）は、**単一語義でも必ず
`senses` 配列（要素1つ）に入れる**。`meaning` などをトップレベルに置く旧形式は後方互換として
読めるが、**取り込み側がトップレベルの `linguistic_features` を保持しない**ため使わない
（`senses` の中身は丸ごと保持される）。`entry_score`・`entry_notes`・`confidence`・`notes` は
語全体のメタなのでトップレベルに置く。

## 調査手順（1語ずつ）
1. 表層形と読みから語を特定する。曖昧な語（同名の作品・人物が複数あるなど）は WebSearch で確定し、
   どれを採ったか `notes` に書く。
   - **同じ表層形で複数の異なる語義（同名の作品・人物・団体等）が無いか毎語 WebSearch で確認する**
     （例:「ピーターパンシンドローム」は通俗心理学の用語と、同名の男性アイドルグループの両方が存在する）。
   - **複数の語義が見つかったら `senses` に複数の要素を並べる**（同音異義語対応・Issue 41）。
     取り込み側は語義ごとに `word_sense` を組み立てる。語の読みが語義で共通なら各 `sense` の `reading`
     は省略してよい（語義で読みが変わるときだけ入れる）。慣用度の高い語義から順に並べる。
2. **立項判定（`entry_score`）**: guidelines の収録4原則（公然性・妥当性・慣用性・同定可能性）に
   照らして 1〜5 のスコアを**全語に**付ける。
   - 5=4原則すべて明確に満たす（辞書・百科事典・公式媒体で即確認）／ 4=満たすが裏取りが薄い ／
     3=1原則にグレー（オーナー判断）／ 2=1原則を明確に欠く ／ 1=複数原則を欠く（内輪語・即興造語・検証不能）
   - **3以下には `entry_notes` を必ず書く**（どの原則を・なぜ欠くか）。登録済みの語への低スコアは
     削除候補の指摘になる（削除の最終判断はオーナー）。
   - スコアは「有名さ」ではなく基準への適合度。マイナーでも4原則を満たせば5になり得る。
3. **意味（`meaning`）**: 公開ページのリード文になり得るトーンで、1〜2文・自己完結・「〜。」の常体で書く。
   **1文目で正体を言い切り、2文目で同定に効く補足**（年代・作者・国・提唱者・用途など）を足す。
   承認済みの文体見本:
   - 作品:「三浦しをんによる小説。箱根駅伝を目指す大学生10人の青春を描く。2006年刊。」
   - 出来事:「1773年、イギリスの植民地課税に抗議した北米植民地の人々がボストン港で紅茶を海に投棄した事件。アメリカ独立革命の発端の一つ。」
   - 学術用語:「ベンゼンの二塩化物にあたる有機化合物。防虫剤やトイレの消臭剤に用いられる。」
   - ことわざ:「好きなことは熱心に取り組むので自然と上達するものだ、という意味のことわざ。」
   表記に疑義がある場合（冠称付き・説明句・異体字など guidelines §3 に反する表層形）は
   `notes` で指摘する。
4. **ジャンル（`genre_path`）**: `genre_path` は**必ず長さ3（大→中→小）**にする。
   - `masters.genres` に適切な**小分類があれば、その大→中→小の名前をそのまま使い** `genre_new: false`。
   - 無ければ、`masters.genres` の**既存の中分類を選び、その下に新しい小分類名を1つだけ足して**
     `genre_new: true` を立てる。
   - **大分類・中分類は絶対に新設しない。** 必ず `masters.genres` にある名前をそのまま使う。
     木に無い中分類（例「世界史」）を書くと、既存の分割軸（地域別の `アジア史・東洋史` `アフリカ史` …）と
     二重になって木が壊れる。
   - **小分類は「親の中分類の範囲を細分する分野名」**にする。親と同じ軸である必要はなく、地域軸の
     中分類の下に時代・テーマ・出来事軸の小分類を作ってよい（承認済み実例:
     `北アメリカ史・中央アメリカ史 > アメリカ独立革命`、`アジア史・東洋史 > 古代文明`、`日本史 > 民衆史`）。
   - **同じ小分類名を複数の親の下に作らない**（検索の絞り込みで同名タグが並び、後から統合もできない）。
     既存の小分類に同名があるのに別の親へ新設したくなったら、既存の方に寄せるか `notes` で相談する。
   - 迷ったら新設せず、既存の小分類のうち最も近いものを選び `notes` に迷いを書く。
   - **人物（実在の個人）は「最も強く想起される一分野」の小分類に置く**（guidelines §6）。全業績の網羅は
     諦めてよく、複数分野の万能人でも代表分野を一つ選ぶ（承認済み実例: `レオンハルト・オイラー` =
     `自然科学 > 数学 > 解析学`）。**職業名（`数学者` `作曲家` 等）の小分類は作らない** — それは
     ジャンルではなくエンティティ（`人名`）の情報で、分野の棚が職業の棚に侵食される。
     小分類は業績語・概念語にも再利用できる分野名にする（`解析学` は微分積分にも使えるが `数学者` は使えない）。
     代表分野が割れて決められないときは新設せず `notes` に迷いを書く。
5. **エンティティ（`entity_type`）**: その語が**何であるか**を `masters.entity_types` から選ぶ（→「ジャンルと
   エンティティを混同しない」）。**全語に付ける**（普通名詞も省略しない）。適切な種類が無ければ新しい名前を
   書いてよい（`notes` に理由を書く）。
   - 基本は `〜名` で揃える。承認済みの実例: `作品名` `キャラクター名` `企業名` `グループ名` `地名`
     `文明名` `技術名` `現象名` `物質名` `文書名` `楽曲名` `芸名`。
   - **実在の人物は `人名`**（職業で分けない。`数学者` `作曲家` のようなエンティティは作らない）。
     芸能人など芸名で流通する人物は既存の `芸名` を優先する。
   - `出来事` `発言` `ことわざ` `学説・主張` `俗称症候群` のように語尾が付かない既存名もある。
     **既存にあればそちらを優先**し、細かく分けすぎない（小説・ゲーム・アニメ・美術作品はいずれも
     `作品名`。ただし楽曲は `楽曲名`、歴史的文書は `文書名` が既存）。
6. **品詞（`part_of_speech`）**: `masters.parts_of_speech` から**一字一句同じ表記で**選ぶ。
   収録語の多くは `普通名詞` か `固有名詞`（マスタに「名詞」は無い）。
7. **語種（`word_origins`）**: `masters.word_origins` から選ぶ（複数可）。このマスタは
   **「外来語」ではなく言語名**（`日本語` `中国語` `英語` …）で持つ。
   - **漢字語（音読みの漢語を含む）と和語は `日本語`**。漢語だからといって `中国語` にしない
     （承認済み実例: `天上天下唯我独尊`＝`日本語`、`不完全性定理`＝`日本語`）。
     `中国語` は現代中国語からの外来語（例: マージャン）に限る。
   - カタカナ外来語は**由来言語**で持つ。人名・地名由来はその人物・土地の言語まで遡る
     （承認済み実例: `ヒルベルトプログラム`＝`英語`+`ドイツ語`、`メソポタミア文明`＝`日本語`+`ギリシャ語`、
     `パラジクロロベンゼン`＝`ドイツ語`+`ギリシャ語`）。
   - **混種語は構成要素ごとに複数の語種を並べて表す**（承認済み実例: `ボストン茶会事件`＝`日本語`+`英語`、
     `エルニーニョ現象`＝`日本語`+`スペイン語`）。
   - 由来言語がマスタに無ければ新しい言語名を書いてよい（例 `ギリシャ語`）。近い言語で代用しない。
8. **言語的特徴（`linguistic_features`）**: 表層形と読みを突き合わせ、読み・表記に関わる現象が
   あれば **該当部分ごとに** `{ "name", "target", "target_reading" }` で付ける。
   `name` は `masters.linguistic_features` から**一字一句同じ表記で**選ぶ。定義と具体例の正は
   [`config/linguistic_features_glossary.yml`](../../../config/linguistic_features_glossary.yml)。
   - **毎語、次の3系統を機械的に点検する**:
     1. **複合語の読みの変化**: 構成要素の単独読みと実際の読みを比べる — 連濁（島→ジマ）・
        半濁音化（発→パツ）・促音化（活→カッ）・連声（反応→ハンノウ）・転音（雨→アマ）・
        音韻添加・音韻脱落。
     2. **漢字の読みの当て方**: 熟字訓（上手→ジョウズ）・重箱読み・湯桶読み・当て字・
        古式読み（傘→カラカサ）・特殊読み。
     3. **語形成**: 畳語（人々）・オノマトペ（擬音語/擬態語）。
   - `target` は**表層形の連続した部分文字列をそのまま**、`target_reading` は**その語義の読み
     （カタカナ）の連続した部分文字列**を書く（取り込み側でどちらも部分一致を検証される。
     一致しないと保存できない）。
   - **同じ該当部分に複数の現象が重なったら、特徴ごとに別要素**にする（承認済み実例:
     `活版印刷術` の「活版（カッパン）」に `促音化` と `半濁音化` の2件）。
   - 同じ `target` が表層形に複数回現れ、複数箇所に同じ特徴を付けるときだけ `target_start`
     （表層形先頭からの文字オフセット・0始まり）を付ける。通常は省略してよい（最初の出現に補完される）。
   - **無い語がほとんどなので、無理に付けない**（カタカナ語・アルファベット語は通常対象外。
     承認済み21語中、特徴が付くのは3語）。
9. **別表記（`variants`）と最も一般的な表記**: まず**その語の最も一般的な表記を WebSearch で確定する**
   （辞書見出し・公式表記・用例数などで判断）。その上で、一般に流通する**それ以外の表記**を毎語
   確認して `variants` に記録する。承認済みの実例が示す守備範囲:
   - 略称（`エレクトロニック・アーツ` → `EA`、`Dead by Daylight` → `DbD`）
   - カタカナ語と漢字表記（`ピーターパンシンドローム` → `ピーターパン症候群`）
   - 通称・和訳（`バミューダトライアングル` → `魔の三角地帯` `魔の三角海域`）
   - 冠称付きの正式名・別形（`不完全性定理` → `ゲーデルの不完全性定理`、`ヒルベルトプログラム` →
     `ヒルベルトの計画` `ヒルベルト計画`）
   - 表記ゆれ（`パラジクロロベンゼン` → `パラジクロルベンゼン`。送り仮名・漢字/かなの揺れも）
   各 `variant` は `surface` + `reading` で書く。**表記が変われば読みも大抵変わるので、`reading` は
   原則付ける**（表層形と読みが同一視できるカタカナ語どうし等、明らかに不要なときだけ省略可）。
   - 取り込み対象の表層形（`word_id` に紐づく登録済み `surface`）が**最も一般的な表記と違う**場合は、
     `surface` を変えることはできない（登録時に確定済み）ので、`notes` に「最も一般的な表記は○○」と
     明記し、一般的な表記を `variants` にも入れてオーナーの判断（表層形の訂正）を促す。
   - 別表記が無ければ省略してよい。
10. `confidence` を付ける: `high`=裏取り済みか定番 / `medium`=妥当だが未検証 / `low`=不確実。
    `low` のときは必ず `notes` に「何に迷ったか・何を確認してほしいか」を書く。
