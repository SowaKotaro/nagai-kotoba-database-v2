# グロース戦略 — アクティブユーザー・PV を健康的に増やすための整理

作成: 2026-07-12(オーナーとの戦略ディスカッションを文書化)

技術面(開発サイクル・スケール時の UX・機能追加)とマーケティング面(SEO・LLMO)に加えて、
サイトを育てるために整理すべき項目を「現状評価つき」でまとめる。
実装タスクに落ちるものは [`docs/issues.md`](issues.md) の Issue 番号を参照する。

凡例: **◎ できている** / **△ 一部できている** / **✗ 未着手**

---

## 0. ポジショニング

goo辞書・Weblio・コトバンク等の総合辞書と正面から戦わず、
**「読みの長さ・リズム・文字種で引ける、長い言葉に特化したデータベース」** という独自の切り口を核にする。
これがブランド・キャッチコピー・SEO の狙いクエリ(「◯文字の言葉」「◯◯なリズムの言葉」)すべての土台。

- ◎ 独自の検索軸(読み文字数・モーラ・先頭/末尾文字・文字種パターン・リズムパターン)は実装済み
- ✗ キャッチコピー・サイト説明文としての明文化はこれから(→ §2 の宿題と合わせて)

## 1. コンテンツ戦略(現フェーズの最重要項目)

データベースサイトは**コンテンツそのものがプロダクト**。技術と同列以上に戦略が要る。

- ◎ 収録基準の文書化: [`docs/annotation-guidelines.md`](annotation-guidelines.md)(収録4原則・立項スコア・表記/読みの基準)
- ◎ 増産体制: 一括登録3ステップ + アノテーション・コンソール + Claude Code 連携(Issue 38)で半自動化済み
- ◎ ページ品質の底上げ: 自動リード文(Issue 18)で「散文ゼロのページ」は根絶済み。用例フィールド(Issue 33)は未着手
- △ 「生きているサイト」の見せ方: 新着5件表示・Atom フィードはあるが、収録数の**推移**を見せる場が無い(→ Issue 34 統計ページに統合)
- **数値目標(宿題)**: インデックス解禁目安 = 注釈済み 300〜500 語(確定事項)。月あたりの注釈スループットを計測し、解禁予定日を逆算する

## 2. ユーザー理解とペルソナ

- ✗ ペルソナ・ユースケースの文書化は未着手
- **宿題(コード変更ではないため Issue 化しない)**: 誰が何のために長い言葉を引くのかを書き出す。
  候補: 作詞・ラップ(リズム/母音検索)、言葉遊び・大喜利、クロスワード/パズル作成(文字数/先頭末尾)、
  創作(ネーミング)、日本語学習者・言語学好き(特徴タグ)。
  ユースケースごとに刺さる機能・狙う検索クエリ・SNS での見せ方が変わるため、§1 の収録方針にも反映する

## 3. 計測基盤と KPI(早いほど良い — 過去データは後から取れない)

- ◎ GA4 実装済み(Issue 19。Turbo 対応の page_view 送信、ENV `GA4_MEASUREMENT_ID`)
- ◎ Search Console / Bing の所有権確認メタタグは ENV 対応済み
- ✗ 本番での測定 ID 設定・所有権確認・sitemap 送信は未実施(→ Issue 44)
- ✗ KPI の定義が無い(→ Issue 44 で本文書に追記する)
  - 候補: PV・AU に加えて「新規/再訪比率」「流入チャネル別セッション」「検索(/search・ファセット)利用率」
    「1訪問あたり閲覧語数」「Atom 購読・API 利用」— 数だけでなく**健康さ**を測る

## 4. リテンション(再訪)設計

公開側にアカウント機能を作らない方針のため、**ログインなしで再訪させる仕掛け**が課題。

- ◎ 「今日の一語」(日替わり・決定的)、Atom フィード(新着20件)、共有しやすいパーマリンク(1語1URL)
- ✗ SNS 定期投稿(今日の一語の X 自動投稿 → Issue 47)
- △ ブックマークしたくなる検索 URL: ファセット URL は共有可能だが、狙いクエリ別のランディング(索引 /browse・ジャンル /genres)はあるので当面十分

## 5. 流入チャネルの多様化(SEO・LLMO 以外)

- ◎ SEO 基盤: sitemap / robots / meta・canonical / JSON-LD / ファセットのインデックス方針(Issue 13〜18)
- ◎ LLMO: llms.txt / 公開 JSON API(CC BY 4.0 表記つき) / 自己完結リード文(Issue 24・25・18)
- ◎ シェア導線(X 共有・URL コピー、Issue 30)・静的 OGP 画像(Issue 14)
- ✗ 単語ごとの動的 OGP 画像(→ 既存 Issue 29。収録数百語超えてから)
- ✗ 「引用したくなるページ」= 統計ページ(→ 既存 Issue 34。被リンク獲得の種)
- △ データ引用チャネル: CC BY 4.0 + API + llms.txt の布石は完了。研究・アプリでの利用実績づくりはこれから

## 6. 信頼性・運用(グロースの土台)

- ◎ CI・セキュリティスキャン(rubocop / brakeman / bundler-audit / importmap audit)・テスト基盤
- ◎ HTTP キャッシュ(ETag・expires_in・Rails.cache。Issue 26)— 1万語規模なら現構成で耐える
- ✗ 死活監視・エラー通知が無い(障害に気づく手段がゼロ → Issue 45)
- ✗ DB バックアップが手動(`~/db_backups` → 日次自動化 Issue 46)
- △ fragment cache の残り(browse・genres → Issue 48)
- **コスト見通し(宿題)**: PV 10倍時のサーバ費用モデルは、計測(§3)でトラフィック実測が取れてから

## 7. 法務・ポリシー

- ◎ データライセンス CC BY 4.0(クレジット = サイト名 + URL)を About・llms.txt・JSON API に明示済み
- ◎ 連絡先メールの公開(About)
- ✗ **プライバシーポリシーが無い**: GA4(gtag/Cookie)で利用者情報を外部送信しているのに公表ページが無い。
  改正電気通信事業法の外部送信規律の観点で**公開前に必須**(→ Issue 42。P0)
- △ API・スクレイピングの利用ポリシー: About の記載で最低限はあるが、アクセス急増時のレート方針は未定(必要になってから)

## 8. 持続可能性(運営体制・収益)

- △ 一人運営の律速 = アノテーション作業。Claude Code 連携(Issue 38・39)で「調査しながら全項目入力」から
  「下書きを承認」へ転換済み。**運営キャパ(月あたり注釈語数)の数値化は宿題**(§1 の解禁逆算と同じ計測)
- ✗ 収益方針(広告・寄付・スポンサー)は未定。入れる場合はデザインシステム(紙×墨×朱)との両立を先に設計する。
  当面は判断不要 — トラフィックが立ってから

## 現フェーズ(コンテンツ拡充期)の優先順位

1. **コンテンツ**: アノテーションを進めて解禁目安(注釈済み 300〜500 語)へ — 全施策の前提
2. **法務**: プライバシーポリシー(Issue 42)— 公開の前提条件
3. **公開準備**: インデックス解禁スイッチ(Issue 43)+ 計測運用の立ち上げ(Issue 44)— 解禁と同時に観測を始める
4. **運用土台**: 監視(Issue 45)・バックアップ自動化(Issue 46)— 人気が出る前に整備
5. **スケール系**(コスト最適化・レート制御・収益)は兆しが見えてからで間に合う

計測(Issue 44)だけは「増え始めてから」では過去データが取れないため、解禁前に完了させる。
