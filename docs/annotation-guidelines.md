# アノテーション・ガイドライン（収録基準と調査補助）

単語の立項（収録）とアノテーションの**客観的な基準**。次の2つの目的を持つ（Issue 39）。

1. **Claude Code の基準点**: `word-reading-research`（登録時の表記・読み調査）と
   `word-annotation-research`（アノテーション提案）の両スキルは、この文書を判断の拠り所にする。
2. **オーナーのバイアス防止**: 収録可否・表記・読みの判断が個人の感覚に流れないよう、
   誰が判定しても同じ結論になる基準を明文化する。内輪語の混入や曖昧な表記選定から DB を守る。

---

## 0. 前提: 収録範囲

- **読みが10文字以上**の日本語の単語を収録対象とする（2026-07-06 オーナー確定。`docs/issues.md` 参照）。
- 読みが10文字未満の語は、以降の原則を満たしても収録しない。

---

## 1. 収録の4原則

立項に値するかは、以下の**4原則すべて**で判定する。各原則は独立しているが相互に補完し合う。
例えば「公然性」を満たしても「慣用性」を欠く語（公開はされているが誰も使っていない語）や、
「慣用性」を満たしても「同定可能性」を欠く語（広く使われるが指す対象が曖昧な語）は立項できない。

| 原則 | 判定質問（1行） | 防ぎたい問題 |
|---|---|---|
| 公然性 | 第三者が公開媒体でその語の存在を確認できるか？ | 内輪ネタ・検証不能な語の混入 |
| 妥当性 | 辞書や索引の見出し語になり得る、過不足のない1語か？ | 説明文・不要な修飾による水増し |
| 慣用性 | ある共同体で実際に使われ続けているか？ | 即興の造語・一時的な表現の混入 |
| 同定可能性 | 聞いた第三者が、指す対象を一意に特定できるか？ | 何を指すか分からない曖昧な名称 |

### 1.1 公然性（Publicity）

**誰でも客観的に存在を確認できる語であること。**

- 一般に公開されており、不特定多数が知り得る。
- 辞書・百科事典・書籍・新聞・公式サイト・ゲーム・漫画など、**第三者が確認できる媒体**に存在する。
- 当事者以外でも「その単語が存在する」ことを検証できる。

| 満たす例 | 満たさない例 |
|---|---|
| 天上天下唯我独尊（仏典・辞書に掲載） | 身内だけで使っているあだ名 |
| モンキー・D・ルフィ（公刊漫画の登場人物） | 友人の飼い猫の名前 |
| バミューダトライアングル（百科事典・多数の書籍） | Discord サーバー内だけの呼称 |
| リヴァイアサン（書名・神話上の存在） | 会話中にその場で決めた略称 |

### 1.2 妥当性（Validity）

**一つの語として過不足なく独立して成立していること。**

- 単独で一語として意味を持つ。
- 説明文や不要な修飾を含まない。**冠称・敬称・肩書きは取り除く**（→ §3 表記の選定基準）。
- 文の一部ではなく、辞書などで**見出し語になり得る単位**である。

| 満たす例 | 満たさない例 | 理由 |
|---|---|---|
| プロテスタンティズムの倫理と資本主義の精神 | ドナルド・トランプ大統領 | 冠称（肩書き）が付いている |
| ドナルド・トランプ | 東京にある長い名前の駅 | 語ではなく説明句 |
| 花は桜木人は武士（ことわざ＝1見出し） | とてつもなく大きな | 修飾句の断片 |
| インフォームド・コンセント | 〜のために | 文の一部 |

補足: 書名・曲名・ことわざ・慣用句は全体で1つの見出し語として扱う（「花は桜木人は武士」は
文の形をしているが、ことわざとして辞書の見出しになるため妥当性を満たす）。

### 1.3 慣用性（Conventionality）

**社会において実際に用いられている語であること。**

- 継続的な使用実績がある（一過性のバズで消えた語は慎重に）。
- 一個人が即興で作っただけの造語ではない。
- 一定の共同体や社会で語として**定着**している。共同体は「日本社会全体」である必要はない
  （特定作品のファンダム・専門分野・地域でも、その共同体で定着していればよい）。

| 満たす例 | 満たさない例 |
|---|---|
| インフォームド・コンセント（定着した専門語） | その場で作った造語 |
| リボルビング払い（社会に定着した商用語） | 自分しか使っていない略語 |
| 広く普及したネットスラング・略称 | 投稿1件でしか確認できない表現 |

### 1.4 同定可能性（Identifiability）

**第三者が何を指している語か一意に理解できること。**

- 聞いた人が対象を特定できる。必要以上に曖昧ではない。
- 固有名詞であれば、**一般的な名称**として認識されている。
- 同名の対象が複数ある場合は、意味（語義）の記述で同定を担保する（同音異義語は語義を分ける）。

| 満たす例 | 満たさない例 |
|---|---|
| 東京スカイツリー | 駅前のラーメン屋 |
| バミューダトライアングル | うちの近所の大きな公園 |
| 中臣鎌足 | 田中さん／うちの先生 |

## 2. 立項スコア（5段階）

4原則への適合度を1〜5で評価する。`word-annotation-research` スキルは全語に `entry_score` を付け、
**3以下**には `entry_notes`（どの原則を・なぜ欠くか）を必ず添える。コンソールは3以下に朱バッジを出す。

| スコア | 定義 | 扱い |
|---|---|---|
| 5 | 4原則すべてを明確に満たす。辞書・百科事典・公式媒体で即確認できる | 登録可 |
| 4 | すべて満たすが、一部の裏取りが単一ソースなど確認が薄い | 登録可（根拠を notes に） |
| 3 | 1つの原則にグレーがある（慣用性が新しすぎる、同定に文脈が要る 等） | **オーナー判断** |
| 2 | 1つの原則を明確に欠く、または2つ以上がグレー | 原則見送り（削除候補） |
| 1 | 複数の原則を明確に欠く（内輪語・即興造語・検証不能） | 立項不可（削除候補） |

- スコアは「語の有名さ」ではなく「**基準への適合度**」。マイナーでも4原則を満たせば5になり得る。
- 既に登録済みの語に低スコアが付いたら、それは**削除候補の指摘**。削除の最終判断はオーナーが行う。

## 3. 表記の選定基準（一般的な表記か）

`surface`（表層形）には、その語の**最も一般的な表記**を採る。

- **冠称・敬称・肩書きを含めない**: ドナルド・トランプ大統領 → ドナルド・トランプ。
- **説明的修飾を含めない**: 見出し語になる最小の単位に切り詰める。
- **異体字・旧字は一般表記へ正規化する**: 廣島 → 広島（固有名詞が旧字を正式表記とする場合を除く。
  例: 內藤 ではなく 内藤、ただし「髙島屋」のように商号が正式に旧字を使う場合はそれに従う）。
- **正式名称と通称**: 社会で最も流通している方を表層形にし、もう一方は別表記（variants）に入れる。
- 中黒・スペース・記号は、原典・辞書の見出しの表記に従う（勝手に足したり削ったりしない）。
- 複数の表記が拮抗する場合は、辞書・百科事典・公式サイトの見出し表記を優先する。

## 4. 読みの基準（正しい読みか）

`reading` は語の**最も一般的な読み**を採る。表記ルールは MeCab との突き合わせに合わせる
（`word-reading-research` スキルと同一のルール）。

- **カタカナ**で書く。長音は「ー」（例: プロテスタンティズム）。
- **助詞は表記どおりのかな**: は→ハ、へ→ヘ、を→ヲ（例: 花は桜木人は武士 → ハナハサクラギヒトハブシ）。
- 促音「ッ」・拗音「ャュョ」は小書き。記号・空白・中黒は読みに含めない。
- 数字は文脈上の一般的な読み（例: 四十七士 → シジュウシチシ）。
- 読みが複数あるときは最も一般的なものを採り、他は候補（alternatives）として残す。
- **でっち上げない**: 確信が持てない読みは Web 検索で裏取りし、独立した2ソースの一致を確認する。
  一致しなければ confidence を下げて候補を併記する。

---

## 5. 知名度・一般性の判定目安（一般に知られている語か）

「知られているか」は主観に流れやすいため、以下の**客観的な手掛かり**で判定する。

- 国語辞典・百科事典（コトバンク、Wikipedia 等）に見出しがある → 強い根拠。
- **互いに独立した複数の情報源**（公式サイト＋報道、書籍＋辞書 等）で確認できる → 十分な根拠。
- 単一の投稿・単一のファンサイトでしか確認できない → 慣用性・公然性のグレー（スコア3以下）。
- 判定に迷う場合は、でっち上げずに confidence を `low` にし、notes に「何を確認してほしいか」を書く。

---

## 6. 言語学的特徴の用語集

連濁・音韻添加・重箱読みなどの定義と具体例は、**`config/linguistic_features_glossary.yml`** が
単一ソース（seed のマスタ名と1対1）。アノテーション・コンソールの特徴選択欄にある
「用語解説」パネルから同じ内容を参照できる。

---

## 運用（どこで使われるか）

- **登録時**: `word-reading-research` スキルが §3（表記）・§4（読み）に従って調査 JSON を作る。
- **アノテーション時**: `word-annotation-research` スキルが §1〜§5 に従い、`entry_score` 付きの
  提案 JSON を作る。コンソールの提案パネルがスコアを表示し、3以下に朱バッジと理由を出す。
- **迷ったら**: 原則に照らして notes に理由を書き、最終判断はオーナー（人間）が行う。
  基準そのものに不備を見つけたら、この文書を更新してからスキルの挙動を変える。
